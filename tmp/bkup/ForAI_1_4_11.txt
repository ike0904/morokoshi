## なぜこれが起きるのか：仮説を訂正します

これまで「ミュート設定のタイミング」を疑っていましたが、コードを見る限り`gme_mute_voice`は`gme_play`より前に設定済みなので、それ自体は問題なさそうでした。改めて考えると、原因はもっとシンプルだと思います。

> **NESの直流カット用フィルターは、全チャンネルを合算した「最終出力」にかかっています。通常再生（全チャンネルが同時に鳴っている状態）では、1パートが新たに鳴り始めても、既に鳴っている他パートに埋もれて変化幅が小さく、目立つほどのフィルター反応になりません。**
> **しかし、チャンネルを1つだけ残してミュートする「個別レンダリング」では、そのチャンネル単体が無音から鳴り始める瞬間が、出力全体にとっての「ゼロからの急激な変化」そのものになってしまい、フィルターが過敏に反応してしまいます。**

これなら、これまでの観測結果すべてと辻褄が合います。

| 観測事実 | この説明との整合性 |
|:---|:---|
| 全チャンネル・全ファイルで起きる | チャンネル単体が鳴り始める瞬間は、どの曲のどのチャンネルにも必ずある現象だから |
| 全OFFなら鳴らない | 鳴る瞬間自体が存在しない（鳴らないチャンネルはそもそも音を合算しない）から |
| Winampでは起きない | Winampは全チャンネル同時再生のみ。**個別チャンネルだけを単独で鳴らす「ソロレンダリング」という処理自体が、このアプリ固有の状況**だから |
| 固定フレーム数で削ると「削りすぎる/ノイズが残る」が両方起きる | チャンネルが実際に鳴り始めるタイミングが、曲・チャンネルごとにバラバラだから（フレーム0で鳴り始める曲もあれば、もっと後の曲もある） |

## 提案：ミュート設定そのもののタイミングを変える

ミュートを「最初から」ではなく、**「全チャンネルが鳴っている、ごく短いウォームアップ期間を経てから」適用する**方式に変えることを提案します。

```python
err2 = gme_lib.gme_start_track(emu, track_idx)

# ① まず全チャンネルON（ミュートなし）のまま、Winampと同じ「通常起動」状態を
#    一定時間(例: 30ms)だけ再生して捨てる。録音はしない。
warmup_samples = int(0.030 * NSF_SR)  # 30ms。要調整(実測では20msで収束)
_dummy_buf = (_ct.c_int16 * (warmup_samples * 2))()
gme_lib.gme_play(emu, warmup_samples * 2, _dummy_buf)

# ② ここで初めてミュートマスクを適用する
for i in range(ch_count):
    gme_lib.gme_mute_voice(emu, i, 0 if i == ch else 1)

# ③ ここから本番の記録を開始（既存のwhileループへ）
```

**狙い**：チャンネルの「鳴り始め」自体を、他チャンネルと混ざった状態（＝Winampと同じ、フィルターが過敏に反応しない状態）の中で済ませてしまい、**ミュートへの切り替えはチャンネルが既に鳴っている状態に対して行う**ことで、フィルターにとっての「急激な変化」を作らないようにします。

---

## トレードオフ（正直に書きます）

この方式でも、**最初の30ms以内に始まる音の「アタック（出だしの瞬間）」は、ウォームアップ期間中に紛れて記録されないため、わずかに削れます**。ただし、これは今までの「振幅で判定して最大8フレーム(133ms)も削る」「固定で1〜2フレーム丸ごと無音にする」と比べると、影響範囲がずっと小さく、かつ**毎回ほぼ一定（30ms）**になるはずです。
