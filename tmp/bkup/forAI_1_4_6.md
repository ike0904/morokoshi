nsfファイル対応 仕様書（検討まとめ版）

このファイルは、当初のforAI.txtに対して行った技術検討の結果を反映した更新版です。
Claude Codeでの実装時は、このファイルを起点としてください。

================================================================
■ 全体アーキテクチャ（決定事項）
================================================================

【再生エンジン】
・game_music_emu（通称libgme）を採用。
　- NSFの全既知拡張音源（VRC6, VRC7, FDS, MMC5, Namco163, Sunsoft 5B/FME-7）に対応。
　- チャンネル個別ミュートAPIが標準で用意されている。
　- C言語インターフェースなのでPython側からはctypesで呼び出す。
・配布方法は、現状のffmpeg.exeと同様に「libgme.dllを実行ファイルと同じフォルダに
　同梱する」形。MinGW-w64でビルドする。
・要検証：実際にMinGW-w64でビルドできるか、早い段階で小さく確認しておく。

【拡張音源について：uPD7756C は不採用】
・uPD7756C はNES/Famicomの拡張音源チップとして存在しない（NEC製の音声合成IC。
　ボイス再生用途と推測され、音楽用拡張音源ではない）。
・対応する拡張音源は VRC6 / VRC7 / FDS / MMC5 / Namco163(N163) / Sunsoft 5B の6種類のみ。

【再生方式：「そのまま再生」ではなく「事前デコード」に変更】
・当初案の「wav変換せずそのまま再生」は撤回し、読み込み時に内部で一括デコードして
　メモリ上のPCM配列として保持する方式に変更する。
・理由：NES音源のチップエミュレーションはffmpegの動画デコードと比べて負荷が桁違いに
　軽く、一括デコードは体感できないレベルの速さで終わる見込み（要実測）。
・このおかげで、以下が「既存のWAV再生エンジンの仕組みをそのまま流用」で実現できる
　可能性が高い（当初は対応不可・グレーアウトを想定していた項目）。
　- 波形表示
　- 自由なシーク（波形クリック、ドラッグ等）
　- Speed / Key / Fine の変更（既存のpython-stretchパイプラインがそのまま使える）
　- テンポ自動検出（既存の自己相関ロジックがそのまま使える）
・PCMの保持形式は int16（float32ではない）。最悪ケース（後述）は現時点では考慮しない
　方針のため、メモリ最適化の優先度は今は高くない。

【チャンネルごとの個別レンダリング】
・「チャンネルON/OFF」を実現するため、チャンネル数ぶん（最大でも13ch程度：基本5ch +
　拡張音源最大8ch〔N163〕）、ミュートマスクを変えながらN回デコードし、チャンネルごとの
　モノラルPCM配列(int16)を個別に保持する。
・再生時は、現在ONになっているチャンネルの配列だけをnumpyで合算（サンプル単位の加算）
　してミックスし、その結果を既存のSpeed/Key/Fine変換パイプラインに流す。
・このアプローチは実例あり（kode54氏のmultidumperというツールが同種の処理を実装し、
　実用的な速度で動作することが確認されている）。
・副産物として以下も同じ仕組みで実現できる。
　- 「未使用チャンネルはグレーアウト」→ チャンネルごとの配列がほぼ無音かどうかで自動判定
　- Solo機能（特定chのみON）→ 「ONのものだけ合算」の仕組みで追加実装ほぼ不要
・チャンネルをトグルした際、波形が0クロスでない位置で途切れることによる「プツッ」という
　ノイズ対策として、数msのフェードを挟む。実装時に忘れず反映すること。

【総再生時間の決定方式：内部状態ハッシュ方式は撤回し、「デフォルト1分＋手動延長」に変更】
・当初検討していた「内部状態ハッシュによるループ自動検出」は、うまくいかないことが
　判明したため撤回。代わりに、ユーザーが必要に応じて手動延長する方式に変更する。

　1. 楽曲読み込み時、まず一律で「総再生時間1分」としてデコードする
　　（チャンネルごとの個別レンダリングも、この1分ぶんに対して行う）。

　2. 1分以内に曲が自然終了した場合（ループしない曲）
　　→ 当初仕様どおり、最低10秒を保証したうえで曲の終わりで再生終了。
　　　総再生時間テキストボックスは編集不可・点滅なし（通常のWAV再生時と同じ扱い）。

　3. 1分時点でまだ鳴っている場合（ループ曲など、自然終了が確認できなかった場合）
　　→ 1分ちょうどの楽曲を読み込んだときと同じ振る舞いで、1分で再生を停止する。
　　→ 総再生時間テキストボックスが編集可能になり、赤点滅する。
　　　・ドラッグ／ホイールで1分単位の増減が可能。
　　　・直接のテキスト入力は不可。
　　　・上限は15分。

　4. 時間が（ドラッグ/ホイールで）変更されたら、読み込まれていない差分時間ぶんだけ
　　追加デコードする。
　　・実現方法：チャンネルごとのエミュレータインスタンスを楽曲を閉じるまで破棄せず
　　　保持しておき、延長時はそこから続きを生成する形にする。チップエミュレーションは
　　　決定論的（同条件なら毎回同じ出力）なため、これは「最初から通しで1本生成した
　　　音声」と完全に同一になり、継ぎ目の不自然さは原理的に発生しない
　　　（最初から「差分だけ録音」する設計を前提とする）。
　　・延長は全チャンネルぶん同時に行う。

　5. 延長後、新たに自然終了が確認できた場合
　　→ 総再生時間をその時間に確定し、テキストボックスは編集不可・点滅なしに戻る。
　　→ 確認できなかった場合は、3.と同様に編集可能・赤点滅のまま、再度延長操作を待つ
　　　（上限15分に達したら、それ以上は延長不可）。

【自然終了の判定方法】
・全チャンネル合算後の音量が、5秒間ほぼ無音（無音とみなす閾値は実装時に調整）で
　あれば「自然終了」とみなす。
・この判定は、最初の1分のデコード時・延長後のデコード時の両方で同じロジックを使う。

【延長時のチャンネル・グレーアウト再評価】
・延長によって新たに「使用している」と判明したチャンネル → 自動でONにする。
・すでにユーザーが手動でOFFにしていたチャンネル（延長前の時点で使用が判明しており、
　ユーザーが意図的にOFFにした場合）→ そのユーザーの選択を維持する（勝手にONへ
　戻さない）。

【総再生時間の保存】
・ユーザーが設定した総再生時間は、楽曲ごとの状態保存に含め、次回読み込み時に復元する。

【総再生時間を短縮した場合の扱い】
・テキストボックスをドラッグ／ホイールで短縮方向に操作した場合、見た目の波形表示・
　再生範囲は短縮されるが、デコード済みのPCMデータ（チャンネルごと）は破棄せず裏で
　保持しておく。再度延長する際に同じ範囲を再デコードせずに済むようにするため。

【補足：最悪ケース（15分・多チャンネル）について】
・この延長操作はユーザーが任意に行うものであり、長尺曲では普通に起こりうる操作として
　扱う（「稀なケースだから考慮しない」という位置づけではない）。int16採用により、
　13ch×15分でも概算1GB弱に収まる見込みのため、特別な追加対応は行わない。

■ 項目別の検討結果
================================================================

・1～3ch、ノイズ、DPCMに対応 → 対応可能（libgme標準機能）
・拡張音源対応（VRC6/VRC7/FDS/MMC5/N163/Sunsoft 5B） → 対応可能（uPD7756Cは対象外）
・各チャンネルのON/OFF → 対応可能（チャンネルごと個別レンダリング方式で実現）

・nsf内の複数楽曲の切り替え → 対応可能（NSF形式が標準でトラック番号を持つ）
・状態の楽曲ごと保存、リセットオールの影響範囲限定 → 当初仕様どおり対応可能
　（既存の保存ロジックをファイル名+トラック番号で拡張）

・波形表示 → 対応可能。事前デコード方式により実現。表示範囲は「総再生時間の決定方式」
　（デフォルト1分＋手動延長）で確定した範囲に追従する。

・wav変換せずそのまま再生 → 撤回。事前デコード方式に変更（上記アーキテクチャ参照）。

・ピッチ(Key)・Fine → 対応可能と見込む（グレーアウト不要）。事前デコード方式により
　既存のstretch処理がそのまま使える可能性が高い。実装して動作確認すること。

・スピード変更 → 対応可能と見込む（x0.5限定ではなく、通常のSpeed範囲で対応できる
　可能性が高い）。同上、実装して動作確認すること。

・テンポ検出 → 対応可能と見込む。事前デコード済みのPCMに対して既存のロジックが
　そのまま使えるはず。

■ 専用表示部の仕様
================================================================

スペアナエリアを削除して代わりに入れることで実現する。
（nsf読み込み時のみ、このエリアを書き換える。スペアナは使えない）
エリアの大きさは変えない。


【エリア内の表示概念図】
--------------------------
123 / 234    タイトルタイトルタイトルタイトルタイトルタイトル
[1][2][3][4][5] [6][7][8][9][0][-][^][\]
--------------------------


【123 / 234（楽曲番号）】
楽曲番号を表す。
左(123)：nsfファイル内に入っている複数の楽曲のうち、現在再生中の曲が何曲目か。編集可能テキストボックス。
右(234)：nsfファイル内に入っている楽曲の総曲数。

楽曲番号テキストボックスは他と同様にダブルクリックやドラッグ・ホイールで編集可能。

nsfモードの時のみ、以下の動作が可能になる。テンキーの1・3も同様。
Shift+[←]　前の楽曲
Shift+[→]　次の楽曲


【タイトル表示：NESM / NSFe で挙動を分岐（新規決定事項）】

お手元のファイルが全てNESM形式だったため、タイトル表示は以下のようにファイル形式で
分岐させる。NESM形式は曲ごとのタイトル情報を持てない（ヘッダーのタイトルはファイル
全体で1つだけ）ため、NSFeを前提とした当初仕様をそのまま適用すると意味をなさない。

  ファイル読み込み時にヘッダーが
    "NESM" → タイトル欄＝ファイル全体のタイトルのみを表示（曲番号の付加なし。
              曲番号は左の 123/234 で分かるため）。ツールチップは表示しない。
    "NSFe" → 当初仕様どおり（曲ごとのタイトル表示、9行ツールチップ、現在曲は赤字）。

以下、NSFeの場合に適用される当初仕様：

現在再生中の楽曲のタイトル。文字数が画面内に収まらない場合、スクロールして表示。

楽曲番号テキストボックス編集中のみ、ツールチップで以下の表示を行う。
　・9行のテキストボックスを表示し、現在再生中の楽曲と前後4曲分のタイトルを表示。現在楽曲のみ赤文字で表示する。
　　テキストボックスの横幅は20文字程度あれば十分。文字数がはみ出してもスクロールや改行はせず、そこで表示を打ち切る。
　　1～4曲目が選択されている場合、常に1～9曲目を表示する。
　　同様に、最後の4曲のうちいずれかが選択されている場合、常に最後の9曲を表示する。


【[1]～[\]（チャンネルON/OFFボタン）】
　チャンネルのON/OFFボタン。形状は、今のA(B)マーカーと同じ。
　[5]と[6]の間に隙間を空ける。

　[1]～[2] パルス波
　[3] 三角波
　[4] ノイズ
　[5] DPCM
　[6]～[\] 拡張音源を使用している場合、それのチャンネルを充てる。

　該当楽曲で全く使用していないチャンネルはグレーアウト
　（チャンネルごとの個別レンダリング結果がほぼ無音かどうかで自動判定する）。
　使用しているチャンネルは、デフォルトでONになる（Aボタンを押したときと同じ状態を維持）。
　トグル式でON/OFFが可能。
　SHIFT+クリックで、該当チャンネルのみON,他はすべてOFF。
　右クリックでリセット（グレーアウト以外全部ON）。
　トグル時の音切れノイズ対策として、数msのフェードを挟む。

　メインキーボードの最上段(1～\)でショートカットが可能。
　この仕様追加に伴い、テンキー操作のメインキー対応は廃止。

■ 未確定・要検証事項（実装に着手する前に確認しておきたいこと）
================================================================

・MinGW-w64でlibgmeを自前ビルドできるか（早期に小さく検証する）。
・チャンネルごとのエミュレータインスタンスを「楽曲を閉じるまで保持」する実装が、
　メモリ・リソース管理上問題なく成立するか（例えば曲の切り替え時に確実に破棄
　されるか等）。
・無音判定の閾値（何秒・どの程度の音量を「無音」とみなすか）は、実際の曲で
　試しながら調整が必要。5秒という値を起点に実装し、誤検出（曲中の静かな間奏等を
　終了と誤判定しないか）が無いか確認する。
・総再生時間テキストボックスのUI上の配置（既存の総時間表示〔右側の00:00.0等〕を
　NSFモード時のみ編集可能・赤点滅の挙動に切り替える想定。新規ウィジェットを追加
　するのか、既存ウィジェットの挙動を条件分岐するのか、実装時に決定する）。
・チャンネルごとの個別レンダリング所要時間のベンチマーク（曲を開いてから再生開始
　までの体感速度、および延長操作時の体感速度に影響）。
